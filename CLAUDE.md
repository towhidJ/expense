## Golden rules

- **Money movements must go through Postgres RPCs** (`process_transaction`, `process_transfer`, `process_loan_repayment`, `process_new_loan`, `process_saving`, `process_bazar_purchase`, `process_inventory_movement`, `update/delete_transaction_with_balance`) — never plain inserts/updates, or `accounts.current_balance` (or `inventory_items.quantity`) silently corrupts.
- **File storage is self-hosted MinIO, not Supabase Storage** (migrated 2026-07-19): the MinIO access/secret key must never reach the browser or the mobile app, so all uploads/deletes proxy through the `minio-storage` edge function (`supabase/functions/minio-storage`), which holds the credential as Supabase secrets and signs S3 requests server-side via `aws4fetch`. Web calls it with `src/lib/minioStorage.js` (`uploadToMinio`/`removeFromMinio`, raw bytes + query-param metadata); mobile calls it with `mobile/lib/minio_storage.dart` (`MinioStorage.upload`/`.remove`, JSON+base64 body — same shape `ai_service.dart` already uses for images, avoiding a new `http` package dependency). Bucket allowlist lives in the edge function: `documents` (attachments/vault/meal receipts — any authenticated user) is `user`-level, `app-manuals` and `app-releases` (APK OTA) are `admin`-level. Downloads stay plain public URLs with no auth — each bucket's MinIO anonymous policy is set to download-only, never write, so a stolen URL can't be used to overwrite files. When adding a new upload feature, call `uploadToMinio`/`MinioStorage.upload` — never touch `supabase.storage` again.
- **Every table is scoped by `entity_id`** (personal/family/business workspaces). `EntityContext.currentEntity` drives all hooks — a missing `currentEntity` in a `useCallback` dep array is the classic bug here (stale data after workspace switch).
- **Migrations are manual**: schema lives in numbered `supabase_migration_v*.sql` files at repo root; each new one must be pasted into the Supabase SQL Editor by the user. Latest: **v44** (admin user delete: dynamically rewrites every FK referencing `profiles`/`auth.users` to ON DELETE CASCADE so `auth.admin.deleteUser` can erase an account with data; paired with admin-users edge fn `toggle_ban`/`delete_user` actions + AdminUsers Block/Delete buttons — both refuse self and other admins; blocked = `ban_duration '876000h'`, `banned` flag surfaced in list_users; requires `supabase functions deploy admin-users`); v43 (drops the stale 9-param `process_transaction` overload left behind when v30 added `p_family_member_id` — with both versions present, PostgREST rejected every 9-argument RPC call with "Could not choose the best candidate function"); v42 (free-tier workspace limit: a free account may have only one `entities` row — the "Personal" one from signup; splits the old single `FOR ALL` RLS policy on `entities` into SELECT/UPDATE/DELETE + a separate INSERT policy whose `WITH CHECK` requires `has_premium()` OR `is_app_admin()` OR the user's current entity count `< 1`; enforced server-side, not just client UX — `Sidebar.jsx`'s "+ New workspace" flow pre-checks the same condition to avoid a raw RLS error and instead offers `/subscription`); v41 = 10 new modules — see Web app map for the per-module table list; the one shared piece of new machinery is `process_inventory_movement(item, type, qty, ...)`, a balance-safe RPC for `inventory_items.quantity` that optionally calls `process_transaction` internally to log the linked money movement, same pattern as Bazar/Utility. Every other new module reuses `process_transaction` directly from the client, exactly like `utility_bills.transaction_id`. `liabilities.min_payment`, `attachments.doc_category/expiry_date/title`, and `saving_heads.interest_rate/tenure_months/start_date` (DPS/FDR maturity fields, folded into the existing Savings module rather than a new one) are additive columns on existing tables; v40 = 3-day free trial: `user_subscriptions.is_trial` flag; `handle_new_user()` now also inserts a 3-day trial row for every signup; `admin_set_subscription(user, 'trial')` grants/resets a fresh 3-day trial; `get_my_subscription()` returns `is_trial`; paid approval via `review_subscription_request` clears `is_trial` and starts the paid term from now rather than stacking onto trial days; v39 = commercial billing: `billing_settings` single-row config, `module_access` free/premium map, `subscription_requests` manual-payment queue with partial-unique trx-id index, `user_subscriptions` per-USER — expires_at NULL = lifetime; RPCs `has_premium`/`get_my_subscription`/`submit_subscription_request`/`review_subscription_request`/`admin_set_subscription`; `subscription_update` notification type; admins-read-all-profiles policy; v38 = rent expansion: partial payments — `rent_payments` unique(unit,month) dropped + charge_amount/charge_note; `rent_revisions` rent-increase history; `unit_tenancies` tenant archive with advance settlement; `rent_unit_expenses`; `rent_due` notification type + `check_rent_due()` folded into `finance_daily_alerts()`; v37 = `recurring_transactions.utility_type` + `run_due_recurring` upserts the month's `utility_bills` row as paid; v36 = hotfix: `log_activity()` trigger reads fields via jsonb; v35 = split/insurance/utility/rent/activity-log tables + subscription/warranty/exchange-rate/stock-expiry columns; v34 = Dena-Paona `liabilities.counterparty`). If a feature "doesn't work", first suspect an unapplied migration.
- **Touch-visible actions**: edit/delete buttons must be visible without hover. Web: `opacity-100 sm:opacity-0 sm:group-hover:opacity-100` (hover-reveal desktop only). Flutter: visible trailing `PopupMenuButton` (⋮), never long-press-only. The user considers hidden actions to be missing features.

## Domain gotchas

- Liability type `loan_given` is a **receivable** (adds to net worth), not a debt. There is no `status` column on liabilities — use `remaining_balance > 0`.
- Person-to-person loans (Dena-Paona, `/lending`) are liabilities rows with `counterparty` set; the Liabilities page filters them out (`!l.counterparty`), same pattern as Bazar's `shop_due` shop khatas.
- Recurring transactions auto-post into `transactions`, so trailing monthly averages (used by `/forecast`) already include them — never add recurring on top of averages.
- Meal stock (`meal_stock_items`) has `expiry_date` — StockTab shows expired / expires-in-7d badges; `addStockItem` in useMealData passes it through.
- Meal Manager (`/meals/*`) is a separate multi-user workspace scoped by `group_id` (NOT entity_id) with membership-based RLS via SECURITY DEFINER helpers; month math ONLY via `get_meal_month_summary` RPC (shared web+mobile — never recompute client-side). `meal_group_members.display_name` is a snapshot because profiles RLS is self-only.
- Cash flow statement counts only account-linked movements (due purchases have no account_id, shown as memo); trial balance equity is the balancing figure.
- DPS/FDR is NOT its own module — it's `saving_heads` (on `/savings`) with `saving_type IN ('dps','fdr')`. A head's current balance = sum of its `savings` deposit/withdraw entries; `interest_rate`/`tenure_months`/`start_date` (v41) turn that into a maturity projection. Before building a new "X tracker" module, check whether Savings, Liabilities, or Assets already model it generically — this one almost shipped as a duplicate `/dps` page before the user caught it.
- Zakat page (`/zakat`) persists its settings in localStorage key `zakat_settings_v1`; nisab from per-vori gold/silver prices.

## Flutter app (`mobile/`)

- Package `com.towhid.expense_tracker`; same Supabase project — URL/key hardcoded in `mobile/lib/config.dart`. Data layer: `lib/app_state.dart` (single ChangeNotifier); one screen per feature in `lib/screens/`.
- Premium enforcement: `AppState.refreshBillingState()` (called after `load()`, user-scoped not entity-scoped, fail-open on error) fills `premiumModules`/`billing`/`subActive`/`isAdminUser`; `isLocked(key)` uses the same module keys as web. Locked More-grid tiles show an orange lock badge and open `premium_screen.dart` (status + pricing + bKash/Nagad copy + trx submit + request history, pull-to-refresh); the Bazar bottom tab swaps to PremiumScreen when locked. Admin work stays web-only.
- Near feature parity with web (16 originals + the 10 v41 modules, ported 2026-07-18), EXCEPT: invoice PDF export stays web-only; Reconcile takes pasted CSV text instead of a file upload (no file_picker dependency); Document Vault captures via camera/gallery (image_picker) rather than arbitrary files. PDFs are image-based (RepaintBoundary → PNG) because the dart pdf package can't shape Bengali.
- v41 port plumbing: `AppState.entityRows/insertEntityRow/updateEntityRow/deleteEntityRow` are the mobile mirror of the web's useEntityTable (raw `Map<String, dynamic>` rows, no model classes for the new tables); `processTransactionId()` = process_transaction that returns the tx id for ledger-row linking; `processInventoryMovement()`/`importTransactionsBulk()` wrap the v41/v31 RPCs; vault docs via `fetchVaultDocuments`/`uploadVaultDocument`/`deleteVaultDocument` (attachments + documents bucket). Savings DPS/FDR maturity fields live on `SavingHead` model + savings_screen head form/card, same math as web.
- Build gotchas: keep `kotlin.incremental=false` in `mobile/android/gradle.properties` when the project and Pub cache sit on different drives ("different roots" crash). A failed `flutter build` leaves the previous APK in place — `adb install` after a failed build silently installs the stale one. Gradle "Could not stat file" state corruption → `flutter clean`.
- OTA updates: bump `version:` in `mobile/pubspec.yaml` (the `+N` is the versionCode) BEFORE `flutter build apk`, then enter that same code in the web admin form.

## UI audit trick (no test suite)

The app is auth-gated with email confirmation ON. To render pages headlessly for UI checks: `npm run dev`, then playwright-core + system Chrome with (1) a fake session JSON (hand-built far-future-exp JWT) in localStorage key `sb-<project-ref>-auth-token` — `AuthContext` only calls `getSession()`, no server verification; (2) `context.route()` intercepting `**/<project-ref>.supabase.co/**` returning mock rows per table from the URL path (`/rest/v1/entities` must return ≥1 entity or `currentEntity` stays null; include embedded join keys like `accounts(name)`; `/rest/v1/rpc/*` → null). When checking horizontal overflow, also test `getBoundingClientRect().left < 0` and ignore the intentionally off-canvas `<aside>`.
